# Раздувание (bloat) и очистка (VACUUM) — практика на MovieLens

## Подготовка: проверяем исходное состояние

### [dead-stats] Мёртвые строки по таблицам

```sql
SELECT
    relname                           AS таблица,
    n_live_tup                        AS живых_строк,
    n_dead_tup                        AS мёртвых_строк,
    round(n_dead_tup * 100.0
        / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS мёртвых_процент,
    last_vacuum,
    last_autovacuum,
    last_analyze,
    last_autoanalyze
FROM pg_stat_user_tables
WHERE relname IN ('ratings', 'tags', 'movies', 'links')
ORDER BY n_dead_tup DESC;
```

Результат — `n_dead_tup = 0` везде: `autovacuum` уже поработал после предыдущих
операций с таблицей.

```
+-------+-----------+-------------+---------------+-----------+---------------------------------+
|таблица|живых_строк|мёртвых_строк|мёртвых_процент|last_vacuum|last_autovacuum                  |
+-------+-----------+-------------+---------------+-----------+---------------------------------+
|movies |87585      |0            |0              |null       |2026-03-11 23:29:17.311842 +00:00|
|ratings|31999901   |0            |0              |...        |2026-03-04 03:26:14.012306 +00:00|
|tags   |2000072    |0            |0              |null       |2026-03-04 03:26:11.911205 +00:00|
|links  |87585      |0            |0              |null       |2026-03-04 03:26:12.703369 +00:00|
+-------+-----------+-------------+---------------+-----------+---------------------------------+
```

### [size] Размер таблиц и индексов

```sql
SELECT
    relname                                        AS таблица,
    pg_size_pretty(pg_relation_size(oid))          AS heap,
    pg_size_pretty(pg_indexes_size(oid))           AS индексы,
    pg_size_pretty(pg_total_relation_size(oid))    AS итого
FROM pg_class
WHERE relname IN ('ratings', 'tags', 'movies', 'links')
  AND relkind = 'r'
ORDER BY pg_total_relation_size(oid) DESC;
```

Исходные размеры:

```
+-------+-------+-------+-------+
|таблица|heap   |индексы|итого  |
+-------+-------+-------+-------+
|ratings|1604 MB|686 MB |2290 MB|
|tags   |114 MB |76 MB  |190 MB |
|movies |19 MB  |3864 kB|23 MB  |
|links  |4464 kB|1936 kB|6440 kB|
+-------+-------+-------+-------+
```

## Шаг 1: отключаем autovacuum на tags

Чтобы он не подчистил раздувание раньше, чем мы успеем его наблюдать:

```sql
ALTER TABLE tags SET (autovacuum_enabled = false);
```

## Шаг 2: создаём раздувание

Обновляем `ts` у всех тегов пользователя 119227 (20 369 строк).
PostgreSQL не перезаписывает строки на месте — создаёт новые версии в heap,
старые помечает как мёртвые (dead tuple):

```sql
UPDATE tags
SET ts = ts + 1
WHERE user_id = 119227;
```

### [dead-tags] Мёртвые строки в tags

```sql
SELECT
    relname                           AS таблица,
    n_live_tup                        AS живых_строк,
    n_dead_tup                        AS мёртвых_строк,
    round(n_dead_tup * 100.0
        / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS мёртвых_процент
FROM pg_stat_user_tables
WHERE relname = 'tags';
```

Результат — 20 369 мёртвых строк, ровно столько сколько обновили:

```
+-------+----------+-------------+---------------+
|таблица|живых_строк|мёртвых_строк|мёртвых_процент|
+-------+----------+-------------+---------------+
|tags   |2000072   |20369        |1.01           |
+-------+----------+-------------+---------------+
```

### [size-tags] Размер tags

```sql
SELECT
    relname                                     AS таблица,
    pg_size_pretty(pg_relation_size(oid))       AS heap,
    pg_size_pretty(pg_indexes_size(oid))        AS индексы,
    pg_size_pretty(pg_total_relation_size(oid)) AS итого
FROM pg_class
WHERE relname = 'tags'
  AND relkind = 'r';
```

Таблица выросла с 190 до 192 MB:

```
+-------+------+-------+------+
|таблица|heap  |индексы|итого |
+-------+------+-------+------+
|tags   |115 MB|77 MB  |192 MB|
+-------+------+-------+------+
```

Прирост +1 MB heap и +1 MB индексов — это усиление записи в действии:
один `UPDATE` 20 369 строк обновил и heap, и `tags_pkey`.

## Шаг 3: VACUUM

```sql
VACUUM tags;
```

Повторяем **[dead-tags]** и **[size-tags]**:

```
+-------+----------+-------------+---------------+---------------------------------+
|relname|n_live_tup|n_dead_tup   |last_vacuum                                      |
+-------+----------+-------------+---------------+---------------------------------+
|tags   |2000072   |0            |2026-03-15 19:12:32.992311 +00:00                |
+-------+----------+-------------+---------------+---------------------------------+
```

```
+-------+------+-------+------+
|таблица|heap  |индексы|итого |
+-------+------+-------+------+
|tags   |115 MB|77 MB  |192 MB|
+-------+------+-------+------+
```

`n_dead_tup` обнулился — мёртвые строки убраны. Но размер остался 192 MB:
`VACUUM` освобождает страницы для повторного использования внутри файла,
но не возвращает место операционной системе.

---

## Шаг 4: VACUUM FULL

```sql
VACUUM FULL tags;
```

Повторяем **[size-tags]**:

```
+-------+------+-------+------+
|таблица|heap  |индексы|итого |
+-------+------+-------+------+
|tags   |114 MB|76 MB  |190 MB|
+-------+------+-------+------+
```

Вернулись к исходным 190 MB. `VACUUM FULL` физически переписал таблицу
и индексы без дыр. Но в отличие от обычного `VACUUM` — держал `ACCESS EXCLUSIVE`
блокировку на всё время выполнения.

---

## Возвращаем autovacuum

```sql
ALTER TABLE tags SET (autovacuum_enabled = true);
```

| Состояние | heap | индексы | итого | n_dead_tup |
|---|---|---|---|---|
| Исходное | 114 MB | 76 MB | 190 MB | 0 |
| После `UPDATE` 20 369 строк | 115 MB | 77 MB | 192 MB | 20 369 |
| После `VACUUM` | 115 MB | 77 MB | 192 MB | 0 |
| После `VACUUM FULL` | 114 MB | 76 MB | 190 MB | 0 |

| Операция | Блокировка | Возвращает место ОС | Можно в продакшне без окна |
|---|---|---|---|
| `VACUUM` | нет | нет | да |
| `VACUUM FULL` | `ACCESS EXCLUSIVE` | да | нет |
